  GitHub 기여도 AI 분석 파이프라인 — 기술 설명서

작성 목적: 본 문서는 capstone2 저장소의 분석 엔진이 무엇을 하는지,

================================================================================
1. 시스템 개요
================================================================================

1.1 한 줄 정의
--------------
본 시스템은 백엔드가 수집한 GitHub 프로젝트 기여 데이터(contributions)를 입력으로
받아, 팀원별 기여 점수·협업 품질 점수·코드 품질 점수를 산출하고,
Google Gemini로 활동/PR/이슈 요약을 생성한 뒤, 결과를 백엔드 API에
다시 저장(POST)하는 오프라인 분석 파이프라인(CLI)입니다.

1.2 역할 분담
--------------
  [백엔드 서버]     GitHub 데이터 수집·저장, 프론트엔드 제공
       ↑ GET /api/projects/{id}/contributions
       ↓ POST /api/projects/{id}/ai-analysis
  [본 파이프라인]   점수 계산, NLP 협업 평가, Gemini 요약, payload 조립
       ↑ data.json (contributions 응답 캐시)
       ↓ ai_analysis_payload.json 등 로컬 산출물

1.3 핵심 소스 파일 (실행에 필요한 5개)
--------------------------------------
  main.py              — 진입점, 3단계 오케스트레이션
  ai_engine.py         — 점수 엔진, BGE 협업 NLP, Gemini 사용자/커밋 요약
  ai_analysis_api.py   — collab_network, PR/Issue Gemini, 백엔드 POST
  requirements.txt     — Python 의존성
  README_AI.md         — 실행 방법

================================================================================
2. 전체 처리 흐름 (3단계)
================================================================================

  python main.py --project-id {N}
       │
       ├─ [1/3] GET .../api/projects/{N}/contributions
       │         → data.json 저장 (팀원별 정량·NLP·정적 분석 블록)
       │
       ├─ [2/3] GitHubInsightEngine.analyze(data.json)
       │         → 점수표(analysis_result.csv/json)
       │         → Gemini 사용자 활동·커밋별 요약
       │         → ai_analysis_payload.json (users + collab_network)
       │         → ai_analysis_pr_issue_summaries.json
       │
       └─ [3/3] (선택) POST .../api/projects/{N}/ai-analysis
                 → api_payload + pr_issue_summaries 를 merge 한 body 전송

옵션:
  --no-post              백엔드 저장 생략 (로컬 검증용)
  --local                API 수집 생략, 기존 data.json 만 사용
  --rebuild-payload-only 점수/Gemini 재실행 없이 payload만 재생성

================================================================================
3. 입력 데이터 구조 (data.json)
================================================================================

백엔드 contributions API 응답은 대략 다음 구조입니다.

  {
    "status": "success",
    "project_name": "owner/repo",
    "total_contributors": N,
    "data": [
      {
        "username": "github_id",
        "1_quantitative_data": {
          "commits", "pull_requests", "issues", "code_reviews",
          "loc_added", "loc_deleted", ...
        },
        "2_nlp_data": {
          "commits": [{ "message", "date", ... }],
          "pull_requests": [{ "pr_number", "title", "body", "comments", ... }],
          "issues": [...]
        },
        "3_static_code_analysis_data": {
          "backend_code_score": 0~100 또는 null
        }
      },
      ...
    ]
  }

주의:
  - PR/Issue는 사용자 블록마다 "그 사람과 연관된 PR"만 포함될 수 있음.
  - pr_number는 저장소 전체 번호이며, 목록은 날짜 최신순 등으로 정렬됨.
  - comments가 비어 있으면 collab_network 상호작용 집계가 0이 될 수 있음
    (수집 단계 이슈이지 분석 엔진 버그가 아닐 수 있음).

================================================================================
4. 사용 모델 및 라이브러리
================================================================================

4.1 협업 텍스트 품질 평가 (임베딩)
----------------------------------
  모델 (1순위):  BAAI/bge-m3  (BGE-M3, 다국어 dense embedding)
  모델 (fallback): intfloat/multilingual-e5-large-instruct
  라이브러리:    sentence-transformers
  용도:          커밋 메시지, PR/Issue 제목·본문·댓글 텍스트를 벡터화하여
                 "성의 있는 협업 서술" 앵커 문장과의 유사도 기반 점수 산출

  앵커 문장(양성/음성) 예:
    - 양성: 설계·테스트·이유·마이그레이션 등이 포함된 상세 설명
    - 음성: "수정 완료", "wip fix update" 등 단편적 메시지

4.2 생성형 AI 요약
------------------
  모델 (기본):   gemini-2.5-flash  (환경 변수 GEMINI_MODEL 로 변경 가능)
  SDK:           google-genai (패키지명 google-genai)
  API 키:        환경 변수 GEMINI_API_KEY (Google AI Studio 발급)

  용도:
    (1) 사용자별 전체 활동 한 줄 요약 (gemini_activity_summary)
    (2) 커밋별 한 줄 요약 (analysis_result_gemini_commits.csv)
    (3) PR/Issue별 한 줄 요약 (pr_summary / issue_summary → POST)

  ※ Gemini 요약 입력은 작성자 텍스트(커밋 메시지, PR/Issue 제목·본문) 위주이며,
    PR/Issue 댓글·리뷰는 collab_score(BGE)에는 쓰이지만 Gemini 요약에는 넣지 않음.

  출력 문체: 개조식 명사형 (~했습니다/합니다/했다 종결 금지)

4.3 통계·이상치
---------------
  scikit-learn:
    - RobustScaler: 정량 특성 스케일링 (이상치 탐지 입력용)
    - IsolationForest: 정량 활동 패턴 이상치 탐지 (팀원 8명 이상일 때만)

4.4 기타
--------
  pandas, numpy — 표 처리·점수 집계

※ commit_summary in contributions JSON 은 백엔드(별도 OpenAI 등) 수집 결과이며,
  본 파이프라인의 Gemini 요약과는 별개입니다.

================================================================================
5. 점수 체계 (핵심)
================================================================================

5.1 최종 점수 (final_score)
---------------------------
  final_score = 0.2 × quant_score + 0.6 × collab_score + 0.2 × static_score

  가중치 의미 (quant:collab:static = 2:6:2):
    - collab 60%: 협업·커뮤니케이션 품질을 가장 중요하게 봄
    - quant  20%: 커밋/PR/LOC 등 활동량
    - static 20%: 정적 코드 분석 점수

  모든 하위 점수는 0~100으로 clip.

5.2 무활동 처리 (0점 기반 정책)
-------------------------------
  과거 "기본 70점" 중립 정책은 제거되었습니다.
  활동·텍스트·코드가 없으면 해당 축은 0점에서 시작합니다.

  - quant_score = 0: commits+PR+issues+reviews=0 이고 loc_added=0
  - collab_score = 0: 분석할 커밋/PR/이슈 텍스트가 하나도 없음
  - static_score = 0: 커밋·LOC 없음, 또는 backend_code_score 없음
  - final_score = 0: 위 무활동 조건 시 가중합과 무관하게 0으로 고정

5.3 정량 점수 (quant_score) — quantitative_score
------------------------------------------------
  하위 지표 7개를 각각 0~100으로 환산 후 가중 평균:

    지표                    가중치
    --------------------    ------
    commits                 0.15
    pull_requests           0.20
    issues                  0.15
    code_reviews            0.20
    log1p_loc_added         0.10
    log1p_loc_deleted       0.10
    loc_per_commit          0.10
                            ----
    합계                    1.00

  loc_per_commit = loc_added / (commits + 1)
  log1p_loc_* = log(1 + LOC) 변환 후 평가

  각 지표 점수 산출 (rate_quant_column):
    - 해당 지표값이 0인 사용자 → 그 지표 점수 0
    - 팀원 수 < 5 (MIN_N_RELATIVE): percentile rank × 100
    - 팀원 수 ≥ 5: MAD(중앙값·절대편차) 기반 robust z-score
      score = clip(50 + 0.6745 × (x - median) / MAD × 15, 0, 100)
    - MAD=0이면 percentile fallback

5.4 협업 점수 (collab_score) — qualitative_score (API 필드명)
-------------------------------------------------------------
  BGE-M3 임베딩 + 규칙 기반 루브릭:

  [커밋 채널]
    - Conventional Commits 형식 (feat:, fix: 등) 부분 점수
    - 앵커 유사도 기반 점수 blend

  [PR / Issue 채널]
    - 제목·본문·댓글 텍스트에 설계/이유/테스트 등 키워드 밀도
    - 앵커 유사도 blend

  채널 결합 가중치 (데이터 있는 채널만 재정규화):
    commit 0.25, pr 0.45, issue 0.30

  텍스트가 한 채널도 없으면 collab_score = 0.

5.5 정적 점수 (static_score)
----------------------------
  backend_code_score 만 사용 (수집 단계에서 복잡도 반영된 최종치).
  별도 complexity 블렌드는 하지 않음 (weight 0).

  - commits=0 이고 loc_added=0 → static = 0
  - 활동은 있으나 backend_code_score null → static = 0
  - backend_code_score 있음 → static = clip(score, 0, 100)

5.6 이상치 탐지 (점수와 별도 플래그)
------------------------------------
  팀원 수 ≥ 8 (MIN_N_ISOF) 일 때 IsolationForest on 정량 7특성.
  is_anomaly = -1 이면 "정량 패턴이 팀과 다름" 참고용.
  final_score 계산에는 직접 반영하지 않음 (skipped 시 전원 정상 처리).

================================================================================
6. 협업 네트워크 (collab_network)
================================================================================

POST users[] 각 항목에 포함. "현재 사용자 → 다른 팀원" 방향 그래프.

  형식 (팀원마다 한 행, 상호작용 없어도 0으로 포함):
  {
    "target_username": "teammate",
    "comment_count": n,      // 상대 PR 일반 댓글
    "review_count": n,       // 리뷰로 판별된 댓글
    "issue_comment_count": n,
    "weight": comment + review + issue_comment 합
  }

  집계 방법:
    - 프로젝트 전체 PR/Issue를 pr_number·issue_number로 인덱스 병합
    - PR owner = user.login 또는 블록 소유자 fallback
    - 댓글 작성자 파싱: "[login]:", "[Review(...) - login]:" 등 문자열 패턴
    - 작성자가 분석 대상 사용자이고, owner가 다른 팀원일 때만 카운트

  ※ comments 배열이 비어 있으면 weight는 모두 0 (데이터 한계).

================================================================================
7. Gemini 요약 상세
================================================================================

7.1 사용자 활동 요약 (run_gemini_activity_summaries)
----------------------------------------------------
  대상: analysis_result.csv 의 gemini_activity_summary 열

  Gemini 입력 (collect_activity_text_bundle):
    - 커밋 메시지, PR/Issue 작성자 제목·본문만 포함
    - PR/Issue 댓글·리뷰 코멘트는 제외 (collab_score NLP와 입력 범위가 다름)

  커밋 수 ≤ 25 (GEMINI_COMMIT_CHUNK_SIZE, 기본 25):
    - 1회 API 호출로 overall + commits[] JSON 동시 생성

  커밋 수 > 25:
    - [C 방식] 25개씩 청크로 커밋별 summary만 요청 후 merge
    - overall은 PR/이슈/커밋 맥락만 넣은 별도 1회 요청
    - 청크 실패 시 해당 커밋은 메시지 앞부분 fallback

  상한: 사용자당 최대 GEMINI_MAX_COMMITS_PER_USER (기본 80) 커밋만 요약

7.2 PR/Issue 요약 (run_gemini_pr_issue_summaries)
-------------------------------------------------
  - 프로젝트 전체 PR/Issue 카탈로그 구축 (번호 기준 dedup)
  - Gemini 입력: 작성자 제목·전처리된 본문만 (댓글·리뷰 제외)
  - GEMINI_PR_ISSUE_CHUNK (기본 8)개씩 배치 요청
  - chunk 간 GEMINI_INTER_CHUNK_SLEEP_SEC (기본 1초) 대기
  - chunk 실패 시 PR/Issue 단건 재요청 (B)
  - JSON 파싱 실패 시 최대 GEMINI_JSON_PARSE_RETRIES (기본 2) 재시도 (A)
  - 호출 전 _verify_pr_issue_summary_parser() smoke test (파서 회귀 방지)
  - 최종 실패 시 title/body 기반 fallback 요약

7.3 안정성 메커니즘
------------------
  - 429 / quota: 지수 백오프 재시도 (GEMINI_MAX_RETRIES, 기본 4)
  - JSON 파싱 실패: 재프롬프트 재시도
  - API 키 없음: Gemini 단계 skip, placeholder 문자열

8. 백엔드 POST payload
================================================================================

  merged_body = {
    "users": [
      {
        "username": "...",
        "quantitative_score": quant_score,
        "qualitative_score": collab_score,
        "final_score": final_score,
        "collab_network": [ ... ]
      },
      ...
    ],
    "pull_requests": [
      { "pr_number", "title", "pr_summary" },
      ...
    ],
    "issues": [
      { "issue_number", "title", "issue_summary" },
      ...
    ]
  }

  로컬 파일 분리:
    ai_analysis_payload.json          → users (+ collab_network)
    ai_analysis_pr_issue_summaries.json → pull_requests, issues

  POST 시 두 JSON을 dict.update 로 합침.
  ※ ai_analysis_payload.json 만 보면 PR 요약이 없어 보일 수 있음 (정상).

11. 설계상 한계·주의사항 (발표 시 언급 권장)
================================================================================

  1) 소규모 팀(3~4명): quant 상대 비교가 percentile 위주라 활동량 차이가
     quant에 덜 드러날 수 있음. collab 가중 60%가 순위에 큰 영향.

  2) collab_network는 댓글/리뷰 원문 데이터에 의존. 수집 API에 comments가
     없으면 0만 나옴.

  3) Gemini는 호출 수·시간·quota에 비례. 대규모 OSS(50명+, 100PR+)는
     수십 분 소요 가능.

  4) 본 저장소는 CLI 배치 분석. 프론트 "분석 시작" 버튼 1회 연동은
     별도 통합 API(백엔드 또는 FastAPI 래퍼)가 필요.

  5) PR 번호가 사용자별 목록에서 띄엄띄엄인 것은 저장소 전체 번호 중
     해당 팀원 관련 PR만 포함되기 때문이며 오류가 아님.

================================================================================
12. 용어 대응표 (API ↔ 내부)
================================================================================

  quantitative_score  ←  quant_score
  qualitative_score   ←  collab_score  (협업 NLP 품질; 네트워크 횟수와 별개)
  final_score         ←  weighted linear combination
  pr_summary          ←  Gemini PR 요약 (POST)
  gemini_activity_summary ← CSV only (사용자 전체 활동 한 줄)
